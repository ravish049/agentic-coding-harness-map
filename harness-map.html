AI Coding Agent Harness Map

Anonymized case study. Mapped against 3 industry frameworks (CNCF four pillars · five-layer harness · three loops) and 5 sources. Diagram shows a working setup, where it scores well, and where 4 real gaps land.
Context: extracted from a multi-tenant, RLS-based SaaS in active development. A small team plus Claude Code as the coding harness, ~11 specialist subagents, ~20 pre-commit + CI gates. Product, customer, and internal names are removed; the workflow structure is real.

Evidence base (5 sources, not 1)

  • codagent.beehiiv — Slot Machines & Safety Nets · 3-loop framing
  • Faros — Harness Engineering 2026 · 5-layer harness; CNCF 4-pillar mapping
  • Fowler — Spec-Driven Development (Kiro / Spec Kit / Tessl) · 3 SDD levels: spec-first / spec-anchored / spec-as-source
  • GitHub Copilot cloud agent + SWE-CI benchmark · background agents + long-term maintenance
  • alecnielsen/adversarial-review + Claude × Codex cross-review · model-family blind spots are real (3 critical bugs found in Claude Code's own source)

Why look past any one article

  • The Beehiiv piece nails the 3-loop shape but undersells observability and background execution.
  • Faros / CNCF frame the same idea as platform engineering (golden paths + guardrails + safety nets + manual review). That lens caught the observability gap.
  • Spec-driven dev (Kiro / Tessl) is the strict version of a "spec-first" gate. Knowing the taxonomy stops you from over-investing — most projects need spec-first, not spec-as-source.
  • SWE-CI benchmark explicitly measures codebase maintainability over 233 days, not per-PR correctness. That validates the scheduled-cleanup direction.
  • Cross-model review has concrete evidence, not vibes — GPT found 3 bugs Claude missed when reviewing Claude Code's own source.

The 4 real gaps — where each fits, what it does, what it costs

GAP 1Scheduled cleanup / drift agent

Where it fits
Outer loop · async lane · nightly or weekly GitHub Action cron, not PR-driven.
What it does
Re-runs the multi-expert epic review across active epics; checks docs for drift; greps for known pattern regressions; opens a single tracking issue (not 19 PRs) per week with prioritized findings.
Why now
SWE-CI direction + Faros: per-PR correctness is mostly solved, codebase decay is the next frontier. The gates already exist — this agent runs them on a schedule against the whole repo, not the diff.
What it does not do
Does not open refactor PRs autonomously (too risky for billing/RBAC paths). Findings → tracking issue → human triages.
Existing primitive
Debt ledger + epic review skill. Both already exist; missing piece is the cron.
Effort: ~½ story (one workflow file + one tracking-issue template). Output: weekly debt report.

GAP 2Cross-model reviewer on money / security paths

Where it fits
Inner loop · review tier · gated to a security-critical subtree (money + tenant-isolation paths).
What it does
After the primary reviewer agent (Claude) passes, fire a non-Claude reviewer (Codex/GPT, or Copilot's review-with-different-model) on the same diff. Different training distribution → different blind spots. Evidence: GPT-5.4 caught 3 critical bugs in Claude Code's own source (return code logic, terminal injection, path double-application).
Why scoped
Cross-model review is expensive in tokens AND review cycle time, so it’s reserved for the highest-stakes changes.
Existing primitive
Copilot already reviews PRs (different model family). Gap is making it mandatory on money paths and surfacing its findings into the merge gate.
Effort: ~½ story (1 workflow file + path-scoped check + merge gate). Trial Copilot's cross-review API before adding a 2nd vendor.

GAP 3Agent observability — the 5th harness layer

Where it fits
Cross-cutting · sidecar to every story. The Faros 5-layer model lists this explicitly; most setups have 4 of 5 strong and skip this one.
What it does
Capture per-story: token cost, wall-clock time, # of bypass uses, # of cross-model findings, rework rate (commits-to-merge), regression count post-merge. Surface as a monthly markdown report.
Why now
Without it you cannot tell whether the operational rules are actually reducing rework or just adding friction. Trust-based rules only work if you measure the cost of the trust.
Existing primitive
Bypass log + git history. Missing: aggregator.
Risk if skipped
Rules accrete forever, nobody knows which gates pay rent. Gate-fatigue is real once N rules accumulate.
Effort: 1 story. Defer until Gaps 1 + 2 land — observability is most valuable once there is more automation to measure.

GAP 4Background draft-PR agent (Devin/Copilot-cloud style)

Where it fits
Async lane · narrow scope: only items tagged auto-safe in the debt ledger (typos, dead-code deletes, dependency bumps with green CI, doc-only fixes).
What it does
GitHub Copilot cloud agent (or Devin) reads tagged debt items, opens draft PRs, runs CI. Human merges. No money, auth, or schema paths — ever.
Why later, not now
This is the riskiest gap. It is also the lowest-leverage until Gap 1 produces a clean prioritized backlog. Without Gap 1 there is nothing safe for Gap 4 to pick up.
Honest take
For a small team, Gap 4 is probably not worth it. Re-evaluate at a larger team and a launched product. Putting it on the map so you can deliberately say no to it.
Effort: large (2+ stories). Recommendation: park.
